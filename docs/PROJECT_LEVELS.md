# Project Levels

Open Tools Starter 使用 A / B / C 三档项目等级来控制复杂度。等级不是身份标签，而是交付边界：先把当前级别做完整，再考虑升级。

## C 级项目：轻量小工具

适合：

- 单页工具
- 文本处理
- 提示词生成器
- 二维码工具
- 小型计算器
- 规则简单、输入输出清晰的小工具

目标：

- 轻量
- 快速发布
- 功能少但完整
- 页面打开即用
- 不引入重依赖

必须包含：

- 现代 UI
- 移动端适配
- 深色模式
- 中文 / 英文切换
- 示例数据
- 一键复制
- README
- GitHub Pages 部署

不强制包含：

- PWA
- 复杂导入导出
- self-test
- 分享链接
- 批处理

## B 级项目：标准实用工具

适合：

- PDF 小工具
- 图片工具
- 文件处理
- 批量重命名
- 字幕工具
- JSON / CSV 工具
- 有明确文件输入输出的实用工具

必须包含 C 级所有内容。

额外必须包含：

- localStorage
- 导入 JSON
- 导出 JSON
- 下载结果文件
- 分享链接
- 错误提示
- 空状态
- `self-test.html`
- `self-test.js`
- `RELEASE_NOTES.md`
- 发布前检查清单
- 隐私说明

控制原则：

- 可以处理文件，但默认必须在浏览器本地处理。
- 不上传用户文件到服务器。
- 不引入后端、数据库或登录系统。
- 所有按钮必须真实可用。

## A 级项目：旗舰项目

适合：

- 长期维护的工具箱
- File Bento
- PDF Desk Lite
- Image Desk Lite
- Prompt Market
- GitHub Galaxy
- 多模块、多场景、多版本维护的前端工具项目

必须包含 B 级所有内容。

额外必须包含：

- PWA
- 离线提示
- 批处理能力
- 工具模块化
- 统一组件库
- 统一设置中心
- 项目健康检查
- preflight 脚本
- SEO / OpenGraph
- 完整错误边界
- 完整中英文文案

控制原则：

- A 级不是“功能越多越好”，而是“长期维护能力完整”。
- 所有工具模块都应可独立理解、独立测试、独立降级。
- 每次发布前必须运行 build、self-test 和 preflight。

## 升级建议

从 C 到 B：

- 先补 localStorage、导入导出、下载结果文件和错误状态。
- 再补 self-test、发布说明和隐私说明。
- 最后补分享链接和发布前检查。

从 B 到 A：

- 先做模块化和统一设置中心。
- 再补 PWA、离线提示、健康检查和 preflight。
- 最后完善 SEO、OpenGraph、错误边界和完整双语文案。

## 不推荐行为

- 为了模板好看而做空按钮。
- 为了炫技引入重型依赖。
- 在 C 级项目中过早加入复杂批处理。
- 在纯前端工具里加入后端、数据库或登录。
- 未经明确说明上传用户文件。
